# DIP 專案演算法開發與協作規範 (AGENTS.md)

本文件定義了使用者（開發者）與 AI 協同開發 `dip_proc.dll` 影像處理演算法的核心流程與規範。在接下來的開發過程中，**所有 AI 代理（Agents）均須嚴格遵守以下流程**。

---

## 1. 撰寫階段 (Writing & Development Stage)

此階段著重於單一演算法的觀念釐清與架構設計，嚴禁在未獲得授權前直接實作程式碼。

### 協作步驟：
1. **選定功能**：使用者每次選定**一個**欲撰寫的演算法函式（一次僅限開發一個，不得多個同時進行）。
2. **知識與原理說明**：AI 必須先詳細解釋該演算法涉及的**數學原理、核心觀念、影像處理工作流 (Workflow)** 等背景知識。
3. **架構規劃與確認**：
   * 等待使用者理解上述概念並回覆「**OK**」之後，AI 才能開始規劃該演算法的函式架構與實作邏輯。
   * 如果使用者提出詢問、調整需求或有疑義，AI 必須針對使用者需求進行解答與調整，**此時絕對不可以直接實作或提供完整程式碼**。
   * **特別協作路徑 (使用者自行編寫)**：若使用者回覆已經「手動/自己寫完成演算法程式」，在驗證其演算法邏輯正確後，AI 須分析專案的架構安排、合理性等資訊（可選擇提出架構可以如何進一步優化與更改的建議），隨後直接跳至 **5. 語法與編譯器驗證** 進行編譯測試並回報結果。若結果正確無誤，即可進入 **2. 測試階段**；若有編譯錯誤，則開始進行 Debug 或分析錯誤原因，並給予改進方向的建議。
4. **開始實作**：直到使用者明確回覆「**開始實作**」之後，AI 始可開始撰寫該演算法的 C++ 程式碼。
5. **語法與編譯器驗證**：
   * 程式碼撰寫完成後，AI 須立即進行專案編譯測試（MSBuild）。
   * **注意：** 此步驟的目的在於確認 C++ 語法正確性、編譯器與 LSP（語法分析器）是否有任何報錯或警告，**非測試演算法實際的影像處理功能**。

---

## 2. 測試階段 (Testing Stage)

此階段著重於單一演算法的整合功能測試與 UI 互動調整。

### 協作步驟：
1. **完整建置**：編譯整個方案項目（C++ DLL 與 C# 前端）。
2. **執行主程式**：運行產出的主程式執行檔 (`DIP.exe`)。
3. **提供測試指引**：針對使用者剛才選定並實作的演算法，AI 必須給予使用者具體的測試建議，包括：
   * **建議測試功能的步驟** (Step-by-step test guide)
   * **驗證目標** (Verification goals)
   * **預期結果** (Expected output / visual changes)
4. **回饋與 UI 微調與設計規範**：
   * **UI 版面規範**：凡是需要使用者手動調整參數、操作（而非直接生成新影像）的演算法功能，C# 前端視窗皆必須採用「上方為影像預覽 PictureBox，下方為參數調整操作 Panel」的整合式版面。
   * **即時預覽 (Real-time Preview)**：當使用者操作滑桿 (Slider) 或其他參數控制項時，必須能即時調用 C++ 演算法進行影像運算，並即時更新預覽的影像，提供使用者時時預覽的回饋體驗。
   * 根據使用者的實際測試反饋，AI 協助微調 C# 前端 UI（如控制項配置、文字標籤、滑桿數值範圍等）。
   * **重要規則：** 若測試中發現需要調整或修改「**演算法核心邏輯**」，必須立刻中斷測試，**退回到「1. 撰寫階段」**，從原理講解與架構重新開始，並等待使用者回覆 OK 才能繼續進行。
5. **彙整變更文件**：
   * 當使用者測試並回報輸出結果與預期相同（確認功能無誤）後，AI 須統整所有變更資訊，並彙整更新 `README.md` 及 `AGENTS.md` 檔案，確保專案文件即時反映最新狀態。

---

## 3. 最終實際部署環境測試 (Final Deployment Environment Testing)

當所有演算法函式皆完成實作後，才能進入此最終部署測試階段。**在所有功能未全部完成前，絕對不可以進入此階段。**

### 執行準則：
1. **確認完整實作**：確認 `dip_proc.cpp` 中的所有 stub 函式皆已成功實作完畢。
2. **C# 載入確認**：確認 C# 專案能順利載入 `dip_proc.dll` 且沒有任何 P/Invoke 簽章不符之錯誤。
3. **無損編譯**：確認整個專案建置皆通過，達到 **`0 Error` (零錯誤)**。
4. **精簡部署環境驗證**：
   * 模擬最終實際部署環境，確保該環境**僅包含且必須包含**以下三個項目，不能有其他任何原始碼、專案檔或暫存快取：
     * `DIP.exe` (C# 執行檔)
     * `dip_proc.dll` (C++ 導出動態連結庫)
     * `Standard Images/` (標準測試影像資料夾)
5. **部署測試說明**：AI 必須提供使用者如何在這個乾淨的部署環境中，測試整個專案所有功能完整度的指南與預期表現。
6. **問題回退機制**：如果在部署環境測試中，使用者提出任何疑義或需要修改之處：
   * 若涉及**演算法**：立即退回 **「1. 撰寫階段」**。
   * 若涉及**視窗介面 (UI)**：立即退回 **「2. 測試階段」**。
   * 待修改完畢且經過使用者再次確認後，才能回到本階段。

## 4. 已開發演算法與功能模組追蹤 (Developed Modules)

| 演算法功能 | C++ 函式 | 狀態 | 說明 |
| :--- | :--- | :--- | :--- |
| **影像轉灰階** | `encode_gray` | 已實作 | 採用 BT.601 標準公式轉換 |
| **位元平面切片** | `bit_plane_slice` | 已實作 (Consolidated) | 整合式 MDI 預覽與控制項視窗，支援原始權重與二值化放大雙模式。支援實質灰階影像（如 `RGB to Gray` 轉換後之影像），對彩色影像進行 ALERT 攔截阻擋 |
| **統計直方圖數據** | `calculate_histogram` | 已實作 | 支援 3 通道 (BGR) 獨立統計（單通道時僅統計第一通道）。數據由 C# 前端直方圖側邊欄（Sidebar）讀取並動態分層渲染。 |
| **直方圖等化** | `histogram_equalization` | 已實作 | 支援單通道灰階與三通道 BGR 的獨立累計分佈函數 (CDF) 均衡化，對比度低之影像等化後可顯著增強細節。 |
| **亮度對比與 Gamma 調整** | `adjust_brightness_contrast` | 已實作 (Consolidated) | 整合式 MDI 預覽視窗 `BrightnessContrastGammaForm`，支援**線性模式**（Alpha 對比度 0.1~3.0、Beta 亮度 ±255，含等效 Gamma 動態算式推導）與**非線性 Gamma 冪律模式**（Gamma 0.1~10.0，滑桿 1.0 居中）。曲線預覽圖支援滑鼠點擊平移與拖曳互動操作。含「確定」輸出與「取消」、「重置預設值」按鈕。 |
